# Multi-machine dev sync

This project lives on more than one Mac at a time (laptop, iMac, occasionally a remote Ubuntu server). Files move between Macs via Syncthing. Git history moves via GitHub. These two paths are different and occasionally conflict. This document describes how to stay aligned when switching machines.

## The mental model

- **Syncthing** copies the working tree between machines. It does NOT sync `.git`, `node_modules`, `.venv`, or anything else in `.stignore`.
- **Git** (via `origin` on GitHub) is the only source of truth for history. If a commit exists only on one machine, it is at risk until pushed.
- **Service-level `.env` files** (`services/*/.env`) are generated by `start-all.sh` from `.env.local` on every run. Treat them as build output, not source.
- **`node_modules` and `.venv`** are machine-local. Install them separately on each machine. Never rely on Syncthing for these.

## When switching machines: the checklist

You are arriving at Machine B. Machine A was the one you last worked on. Assume A has commits that B does not know about yet.

### On Machine A, before you leave

```bash
cd ~/Dev.local/production/SubsBuzz

git status --short          # clean, or only the devctx footer
git stash list              # empty, or known stashes
git log --oneline @{u}..    # list commits ahead of origin
git push origin main        # if any commits are ahead
```

If Machine A has uncommitted work you want to keep, commit or stash first. Anything not on origin is invisible to Machine B.

### On Machine B, when you arrive

```bash
cd ~/Dev.local/production/SubsBuzz

# 1. Pull history
git fetch origin
git status                  # expect "behind origin/main by N commits" plus a pile of working-tree changes from Syncthing

# 2. Reconcile working tree with origin
git stash list              # confirm nothing valuable is stashed
git diff --stat main..HEAD  # confirm no divergent commits
git reset --hard origin/main
git clean -fd               # drop any truly orphan untracked files

# 3. Refresh machine-local dependencies
(cd services/frontend    && npm install)
(cd services/data-server && npm install)
(cd services/email-worker && uv venv && uv pip install -r requirements.txt)

# 4. Start the stack
./start-all.sh              # distributes .env.local, boots postgres + redis + services
```

`start-all.sh` handles the env distribution. Do not hand-copy `.env.local` into `services/*/.env` unless you specifically want to skip the service launch.

## Database migrations

The canonical schema lives at `services/data-server/src/db/schema.ts`. Migrations flow either through incremental SQL (`infrastructure/postgres/migrate.sql`) or through Drizzle's direct push. Pick based on the state of the DB on the machine you just arrived on.

### Check first

```bash
PSQL=/opt/homebrew/opt/postgresql@15/bin/psql
$PSQL -d subsbuzz_dev -c "\dt"
```

### Scenario A: DB already has prior schema

You have run digests on this machine before. Apply the idempotent incremental migration:

```bash
$PSQL -d subsbuzz_dev -f infrastructure/postgres/migrate.sql
```

Every statement in that file uses `ADD COLUMN IF NOT EXISTS` or `CREATE ... IF NOT EXISTS`, so re-running is safe and additive. Pre-existing rows keep `NULL` in any newly-added nullable column.

### Scenario B: DB is empty or missing

The machine is fresh. Bootstrap the full schema from `schema.ts` using Drizzle's push mode:

```bash
cd services/data-server
npm install --no-save drizzle-kit@0.31.10
DATABASE_URL="postgresql://$(whoami)@localhost:5432/subsbuzz_dev" \
  ./node_modules/.bin/drizzle-kit push \
    --dialect postgresql \
    --schema ./src/db/schema.ts \
    --url "postgresql://$(whoami)@localhost:5432/subsbuzz_dev"
cd ../..
```

The project pins `drizzle-kit@0.24.2` in `package.json`, which silently exits with code 1 on a first-run push. The `--no-save` install gives you a working newer binary without changing `package.json`. Once migrations exist in the DB, the pinned version works for day-to-day generate/migrate.

### Verify, either scenario

```bash
$PSQL -d subsbuzz_dev -c "\d digest_emails" | grep hero_image_url
# Expected line starts with:  hero_image_url | text |
```

## Adding a new migration

When you change `schema.ts`, also append an idempotent stanza to `infrastructure/postgres/migrate.sql`. Two reasons:

1. Production and the remote dev server deploy via `deploy.sh`, which pipes `migrate.sql` straight into the docker postgres container. Drizzle's push mode is a local-dev convenience, not the deploy path.
2. Adding to `migrate.sql` keeps Scenario A fast on every other machine you use.

The pattern is always the same:

```sql
-- YYYY-Qn: one-line summary of why this column exists
ALTER TABLE <table>
  ADD COLUMN IF NOT EXISTS <column_name> <TYPE>;
```

Never drop data. Never alter a column in place. If you need a destructive change, write a proper up/down migration under `services/data-server/drizzle` and coordinate the deploy.

## Python services need per-machine venvs

The Python services (`api-gateway`, `email-worker`) do NOT run against global or pyenv Python. `start-all.sh` refuses to start them unless `services/<svc>/.venv/bin/python` exists. Bootstrap once per machine:

```bash
(cd services/api-gateway  && uv venv --python 3.11 && uv pip install -r requirements.txt)
(cd services/email-worker && uv venv --python 3.11 && uv pip install -r requirements.txt)
```

Python 3.11 is pinned to match the Dockerfile. `lxml==4.9.3` (transitive via the email-worker) fails to compile against Python 3.13, so do not let `uv venv` fall back to the system default without `--python 3.11`.

## `.env.local` has the login username baked in

The DATABASE_URL inside `.env.local` contains a literal username, e.g. `postgresql://thomasoneill@localhost:5432/subsbuzz_dev`. Brew Postgres creates a role matching `$(whoami)` at install time, so the username in `.env.local` must match the role on whichever machine is running the stack.

Syncthing carries `.env.local` across machines because the folder has no `.stignore` that excludes it. The simplest way to deal with this when your machines have different logins (e.g. laptop user `thomasoneill`, iMac user `moneill`) is to create a role alias on the machine whose login does NOT match:

```bash
PSQL=/opt/homebrew/opt/postgresql@15/bin/psql
$PSQL -d postgres -c "CREATE ROLE thomasoneill WITH LOGIN SUPERUSER;"
$PSQL -d postgres -c "ALTER DATABASE subsbuzz_dev OWNER TO thomasoneill;"
```

Once the alias exists, `.env.local` works unchanged on both machines and Syncthing can carry it without breaking anything. `REASSIGN OWNED BY moneill TO thomasoneill;` may emit an error about system-owned objects; that is harmless because SUPERUSER lets the role access everything regardless.

A cleaner long-term fix is to add `.env.local` to `.stignore` and let each machine keep its own copy, but that requires coordinating the `.stignore` change so Syncthing does not overwrite one machine's correct copy with the other's during the transition.

## Known gotchas

1. **`psql` is not on the default Bash PATH.** Brew installs it at `/opt/homebrew/opt/postgresql@15/bin/psql`. Interactive shells pick it up via aliases. Non-interactive tool invocations (including Claude Code's Bash) need the explicit path.
2. **Syncthing ignores `.git`.** Do not try to force it. Any attempt to sync `.git` across machines will corrupt the index on one side. History flows through GitHub only.
3. **`services/*/.env` files are generated.** `start-all.sh` overwrites them on every run from `.env.local`. Hand-edits get clobbered. Change `.env.local` instead.
4. **`drizzle-kit@0.24.2` and fresh DBs.** As documented in Scenario B above.
5. **Local dev ports are GCP-constrained.** `5500 / 8080 / 3001` are pinned because the OAuth client has those exact origins whitelisted. Do not change them in `.env.local`. See the main `CLAUDE.md` for the full explanation.
6. **Redis must be installed and running.** `brew install redis && brew services start redis`. The email worker will silently retry forever if Redis is down; watch `logs/email-worker.log` for `Cannot connect to redis://...` if something feels stuck.

## Recovery: I made a mess and want to start clean on this machine

Works if the problem is local-only and origin is in a known-good state:

```bash
git fetch origin
git reset --hard origin/main
git clean -fdx              # NOTE: -x also nukes node_modules. Rebuild after.
(cd services/frontend && npm install)
(cd services/data-server && npm install)
(cd services/email-worker && uv venv && uv pip install -r requirements.txt)
dropdb subsbuzz_dev && createdb subsbuzz_dev    # if DB is corrupt
# then apply scenario B migration above
./start-all.sh
```

Never run this on a machine that has unpushed commits. Push first.
